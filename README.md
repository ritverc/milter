# amavis-milter

Milter-модуль для amavisd на Python 3.10, выполняющий спам-анализ по следующим критериям:

1. **Возраст домена отправителя** — WHOIS-запрос к `whois.nic.ru`, срабатывание при возрасте домена ниже порога
2. **Ложный ответ** — наличие префикса `Re:`/`Fw:` в теме при отсутствии валидного заголовка `In-Reply-To` или `References`
3. **Fuzzy-анализ домена** — определение похожести домена отправителя на известные/ранее встречавшиеся домены (typosquatting)

```

├── milter.py            # Milter-демон (протокол MTA)
├── whois_client.py      # Асинхронный WHOIS-клиент для whois.nic.ru

    ├── __init__.py
    ├── base.py           # Базовый класс проверки
    ├── domain_age.py     # Проверка возраста домена
    ├── false_reply.py    # Проверка ложного ответа
    └── fuzzy_domain.py   # Fuzzy-анализ домена
```

## Конфигурация

Все настройки хранятся в `config.toml`. См. подробные комментарии в файле `config.toml`.

### Триггеры

Каждый триггер определяет:
- `type` — тип проверки (`domain_age`, `false_reply`, `fuzzy_domain`)
- `enabled` — включён/выключен
- `params` — параметры конкретной проверки
- `action` — действия при срабатывании:
  - `header_name` / `header_value` — добавляемый X-header
  - `spam_score_increase` — величина увеличения спам-балла
  - `subject_prefix` — опциональный префикс в теме письма

### Исключения и списки доменов

В параметрах триггеров встречается несколько списков, которые **исключают**
домен или значение из проверки. Они работают по-разному в зависимости от
секции — ниже разобран каждый.

#### `trusted_domains` (в секциях `domain_age` и `fuzzy_domain`)

**Список доменов отправителя, для которых проверка полностью пропускается.**

Если домен отправителя (sender domain) найден в этом списке — триггер не
срабатывает, домен не анализируется. Это «белый список отправителей».

- В `domain_age`: крупные почтовые провайдеры (`gmail.com`, `yandex.ru`, …)
  и корпоративные домены заведомо старые, проверять их возраст бессмысленно.
- В `fuzzy_domain`: легитимные домены, которые не должны помечаться как
  lookalike, даже если похожий typosquat уже накопился в истории (см. ниже).

#### `known_domains` (только в секции `fuzzy_domain`)

**Это НЕ список исключений.** Это **эталонный список** — множество
легитимных «жертв», с которыми сравнивается домен отправителя. Триггер
срабатывает, когда домен отправителя похож на один из `known_domains`
(или на домен из истории).

Назначение противоположное `trusted_domains`:
- `known_domains` — «с чем сравниваем» (эталоны/жертвы);
- `trusted_domains` — «кого не проверяем» (белый список отправителей).

#### Нужно ли указывать оба домена в `trusted_domains`?

**Нет.** В `trusted_domains` указывается **только легитимный домен**,
никогда — typosquat/подделка.

Пример: легитимный `gmail.com`, подделка `gmai.com`.

| Список | Что указываем | Зачем |
|--------|--------------|-------|
| `known_domains` | `gmail.com` | чтобы `gmai.com` был распознан как похожий |
| `trusted_domains` | `gmail.com` | чтобы сам `gmail.com` никогда не помечался |
| (нигде) | `gmai.com` | он детектируется похожестью, в списки не вносится |

Легитимный домен обычно присутствует **в обоих** списках одновременно
(`known_domains` + `trusted_domains`) — и это правильно: в `known_domains`
он защищает от подделок, в `trusted_domains` защищает сам себя. Подделка
`gmai.com` не попадает ни в один список — её задача быть пойманной, а не
внесённой в белый список.

#### `same_tld_skip` (только в секции `fuzzy_domain`)

Булев флаг-исключение. При `true` из сравнения исключаются все эталонные
домены, у которых тот же TLD, что и у отправителя (например `mail.ru` ↔
`yandex.ru` — оба `.ru`, сходство часто benign). Снижает ложные срабатывания,
но может пропустить typosquat в том же TLD.

#### История доменов (`history_file`) и её связь с исключениями

Fuzzy-проверка ведёт динамическую историю встреченных доменов (JSON-файл),
которая объединяется с `known_domains` в эталонное множество. Важно: в
историю домен отправителя **добавляется только если триггер НЕ сработал**.
Домен, распознанный как lookalike, в историю не попадает — иначе однажды
пойманный typosquat сам стал бы эталоном и мог бы маскировать будущие
подделки или ложно помечать легитимный домен (когда `gmail.com` сравнится с
записанным `gmai.com`).

### Вырезание существующих префиксов темы

При повторной обработке письма (реинъекция, пересылка обратно через milter и
т.п.) префиксы темы могли бы накапливаться, в том числе «спрятанные» за
префиксами ответа/пересылки:

```
Re: [LOOKALIKE] Re: [LOOKALIKE] Re: [LOOKALIKE] Re: Hello
```

Чтобы этого избежать, при срабатывании правил milter **вырезает все** (не
только ведущие) вхождения нового (сработавшего) префикса из темы — включая те,
что прикрыты `Re:`/`Fw:` — и лишь затем добавляет новый префикс ровно один
раз. Таким образом тема всегда соответствует текущей оценке правил и не
дублирует префиксы.

Поведение управляется параметром в секции `[milter]`:

```toml
[milter]
strip_existing_prefixes = true   # default: true
```

Установите `false`, чтобы вернуть прежнее поведение (всегда добавлять префикс
без вырезания существующих).

Пример:
- Тема до обработки: `Re: [LOOKALIKE] Re: [LOOKALIKE] Re: запрос`
- Срабатывает триггер `fuzzy_domain` → префикс `[LOOKALIKE]`
- Без вырезания: `[LOOKALIKE] Re: [LOOKALIKE] Re: [LOOKALIKE] Re: [LOOKALIKE] Re: запрос`
- С вырезанием: `[LOOKALIKE] Re: Re: Re: запрос`

### Группы (комбинированные правила)

Группы объединяют несколько триггеров и срабатывают по условию:
- `mode = "all"` — все перечисленные триггеры должны сработать
- `mode = "any"` — хотя бы один триггер должен сработать
- `mode = "majority"` — больше половины триггеров должны сработать

Действия группы применяются **дополнительно** к индивидуальным действиям триггеров.

## Интеграция с amavisd

В `amavisd.conf`:

```perl
# Включить milter-подключение
$interface_policy{'8899'} = 'AMAVIS-MILTER';
$policy_bank{'AMAVIS-MILTER'} = {
    protocol => 'AM.PDP',
    inet_acl => [qw(127.0.0.1)],
};
```

В Postfix `main.cf`:

```
smtpd_milters = inet:127.0.0.1:8899
milter_default_action = accept
```
